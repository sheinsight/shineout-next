针对组件属性(prop)的变更，分析影响范围并执行回归性测试。

使用方式：`/prop-regression <属性名>` 或 `/prop-regression <组件名>.<属性名>`

参数说明：
- 参数可以是单独的属性名（如 `bind`），会全局搜索
- 也可以是 `组件.属性` 格式（如 `Select.bind`），缩小搜索范围

步骤：

1. **解析参数**：从 $ARGUMENTS 中提取目标属性名（和可选的组件名）

2. **定位属性定义**：
   - 在 `packages/hooks/src/components/` 和 `packages/base/src/` 中搜索该属性的 TypeScript 类型定义（interface / type）
   - 记录属性的类型签名、所属接口名、定义文件路径

3. **追踪属性消费链路**：
   - 从类型定义出发，搜索该属性在 hooks、base、shineout 三个 package 中的实际使用位置
   - 重点关注：属性值被读取、解构、传递、做类型断言或调用方法（如 `.map()`, `.length`, `.indexOf()` 等）的地方
   - 输出完整的消费链路（从 props 传入 → hooks 处理 → 组件渲染）

4. **分析影响面**：
   - 列出所有直接使用该属性的组件（通过 grep `__example__` 目录和组件源码）
   - 列出所有间接受影响的组件（通过共享 hook 或 context 传递）
   - 按影响程度分级：🔴 直接使用 / 🟡 间接依赖 / 🟢 无关

5. **检查防御性处理**：
   - 检查属性在消费侧是否有充分的类型守卫（如 `Array.isArray()`, `typeof` 检查）
   - 检查是否对 `undefined`、`null`、非预期类型做了兜底
   - 如果发现缺失防御性处理，输出具体的风险点和文件位置

6. **查找相关测试**：
   - 在 `packages/shineout/src/*/__test__/*.spec.tsx` 中搜索与该属性相关的测试用例
   - 在 `packages/shineout/src/*/__example__/` 中搜索使用了该属性的示例文件
   - 标注哪些有测试覆盖、哪些缺失

7. **执行现有测试**：
   - 对所有受影响的组件，运行其已有的测试用例：
     ```
     pnpm test --filter shineout -- --testPathPattern="<component>"
     ```
   - 汇总测试结果（通过/失败/跳过）

8. **生成回归报告**：输出一份结构化报告，格式如下：

```
## 属性回归分析报告：<prop-name>

### 📋 属性定义
- 类型：`<type>`
- 定义位置：`<file:line>`
- 所属接口：`<interface-name>`

### 🔗 消费链路
1. <file:line> - <描述>
2. <file:line> - <描述>
...

### 📊 影响面分析
| 组件 | 影响级别 | 使用方式 | 测试覆盖 |
|------|---------|---------|---------|
| Form | 🔴 直接 | fieldset-context.ts:32 | ❌ 无 |
| Select | 🔴 直接 | 通过 Form 上下文消费 | ✅ select.spec.tsx |
| ...  | ...     | ...     | ...     |

### ⚠️ 风险点
- [ ] <file:line> - <具体风险描述>

### 🧪 测试执行结果
- ✅ <component>: X passed
- ❌ <component>: X failed (失败原因)

### 📝 建议
- 需要补充的测试场景
- 需要加固的防御性处理
```

注意事项：
- 搜索属性时注意区分同名属性（如 `name` 在不同组件有不同含义），通过上下文和接口归属区分
- 对于在 hooks 层定义的公共属性（如 `bind`, `name`, `rules`），影响面通常是所有表单组件，需要全量扫描
- 如果发现防御性处理缺失，在报告中明确标注，但不自动修复（由开发者决定）
- 测试失败时输出具体的错误信息，方便定位
